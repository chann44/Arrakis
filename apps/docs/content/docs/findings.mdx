## What is a finding?

A **finding** is a detected risk in a specific package version. It can be:

- A known CVE/advisory from OSV, GHSA, or NVD
- A supply chain signal (typosquat, install script, dependency confusion, etc.)
- An AI analyzer result

Every finding is tied to:
- A specific package (`lodash@4.17.20`, `requests@2.27.0`)
- A specific advisory (`CVE-2021-23337`, `SUPPLY-CHAIN|TYPOSQUAT`)
- One or more scans that detected it

## Finding schema

```json
{
  "id": 12345,
  "scan_run_id": 789,
  "repository": "acme/api",
  "package_name": "lodash",
  "manager": "npm",
  "registry": "npmjs.org",
  "version_spec": "^4.0.0",
  "resolved_version": "4.17.20",
  "advisory_id": "GHSA-35jh-r3h4-6jhm",
  "aliases": ["CVE-2021-23337"],
  "title": "Command Injection in lodash",
  "summary": "lodash prior to 4.17.21 is vulnerable to Command Injection...",
  "severity": "high",
  "fixed_version": "4.17.21",
  "reference_url": "https://github.com/advisories/GHSA-35jh-r3h4-6jhm",
  "sources": ["osv", "ghsa"],
  "status": "open",
  "first_seen_at": "2026-01-15T10:00:00Z",
  "last_seen_at": "2026-04-10T02:00:00Z"
}
```

## Querying findings

**All findings for a scan:**
```text
GET /v1/scans/{scanID}
```

**All findings across all repos:**
```text
GET /v1/findings
```

Query parameters:

| Parameter | Description |
|---|---|
| `severity` | Filter by severity: `critical`, `high`, `medium`, `low` |
| `status` | Filter by status: `open`, `resolved` |
| `manager` | Filter by package manager: `npm`, `pip`, `go` |
| `repo_id` | Filter by repository |
| `page` | Pagination |
| `limit` | Results per page (default 50) |

## Finding status

| Status | Description |
|---|---|
| `open` | Active finding, not yet addressed |
| `resolved` | Manually marked as resolved |

A finding marked `resolved` will be re-opened (`open`) if it is detected again in a subsequent scan.

<Callout type="info">
  "Resolved" in Arrakis means manually acknowledged — it does not necessarily mean the vulnerability was patched. Use it to indicate findings you've triaged and accepted, or that are not applicable to your use case.
</Callout>

## Deduplication across scans

The same CVE in the same package version detected across multiple scans is tracked as a single `RepositoryFindingOccurrence`:

- `first_seen_at` — Timestamp of the first scan that found it
- `last_seen_at` — Timestamp of the most recent scan that found it

This means your findings list doesn't grow unboundedly with each scan. You see one entry per unique `{package}@{version}/{advisory_id}` combination, with timestamps showing how long it's been present.

## Push findings to issue trackers

Create an issue for a finding directly from Arrakis:

```text
POST /v1/integrations/{provider}/issues
Body: { "finding_id": 12345 }
```

Supported providers: `github`, `jira`, `linear`

See [Integrations](/docs/integrations) for setup details.

## Findings and the dependency graph

A finding on a **transitive** dependency (depth 2–4) still appears in your findings list. The `version_spec` field will show the version specifier from the parent package that pulls it in — not your own `package.json` specifier — so you can trace the dependency chain.

To remediate a transitive finding, you typically need to either:
1. Update the direct dependency that brings in the vulnerable transitive package
2. Override the transitive package version in your lockfile
3. Replace the direct dependency entirely
