## Why supply chain analysis?

A CVE is only filed after a vulnerability is discovered and reported. Supply chain attacks — typosquatting, dependency confusion, malicious install scripts — can be executing on your infrastructure months before a CVE exists.

Arrakis's supply chain analyzer detects behavioral risk signals in package metadata and scores them independently of advisory databases. A package with no CVEs can still get a `critical` finding if it exhibits enough risk signals.

## How it works

`addSupplyChainFlags()` runs against all **direct dependencies** after the dependency graph is built. It reads package metadata that was already resolved during the sync phase — no additional network calls are made during scanning.

For each package, signals are evaluated and a cumulative risk score is computed. If the score exceeds a severity threshold, a finding is created with `advisory_id: SUPPLY-CHAIN|{SIGNAL_NAMES}`.

## Risk signals

### Typosquatting

Detects packages with names very similar to popular, widely-used packages.

**Score: +30** (or **+40** if edit distance is exactly 1)

The analyzer computes string edit distance between the package name and a list of well-known packages. An edit distance of 1–2 with high confidence is flagged.

**Example:** `lodahs` vs `lodash`, `reqeusts` vs `requests`

### Install scripts

npm packages that define `preinstall`, `install`, or `postinstall` scripts in `package.json` can execute arbitrary code when installed. This is a common attack vector.

**Score: +25**

The metadata field `has_install_script` is set during the registry resolution phase by inspecting the package's `package.json` in the npm registry.

<Callout type="warn">
  Not all packages with install scripts are malicious — many legitimate packages (node-gyp, esbuild, etc.) use them for native compilation. Combine this signal with others when assessing risk.
</Callout>

### Dependency confusion

Detects packages whose names suggest they might be internal/private packages published to a public registry — a common attack vector where an attacker publishes a malicious package with the same name as an internal package, relying on the resolver to pick the public version.

**Score: +35**

Signals include:
- Package name matches patterns like `@company/internal-*`, `internal-*`, `private-*`
- No public repository or homepage URL
- Very low download count relative to name prominence
- First published very recently

### Known malware

Checks the package name and version against a curated list of packages confirmed as malicious and reported to package registries.

**Score: +50**

This is the highest-weighted signal. Any match here will result in a `critical` finding regardless of other signals.

### Version anomaly

Detects unusual version release patterns that may indicate a compromised package or a supply chain substitution attack.

**Score: +30**

Patterns detected:
- Unexpectedly large version jump (e.g., 1.0.0 → 9999.0.0)
- Version released significantly faster than the package's historical cadence
- Multiple versions published in a very short time window

### Missing repository

Packages without a public repository URL are harder to audit and more likely to be malicious or abandoned.

**Score: +10**

### Abandoned with vulnerability

A package that hasn't been updated in over a year and has at least one known CVE is flagged. An active maintainer would typically patch known vulnerabilities.

**Score: +15**

### Single maintainer

Packages maintained by a single person have higher risk if that account is compromised or the maintainer disappears.

**Score: +5**

This signal is weighted low because many high-quality packages are solo-maintained. It contributes to the score only when combined with other signals.

### Missing lockfile (repo-level)

If the repository has no lockfile (`package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`, `poetry.lock`, etc.), dependency resolution is non-deterministic and supply chain attacks via version ranges become easier.

**Score: +20** (applied once per scan, not per package)

## Severity thresholds

| Threshold | Severity |
|---|---|
| Score ≥ 50 | Critical |
| Score ≥ 30 | High |
| Score ≥ 15 | Medium |
| Score < 15 | Not reported |

## Finding format

Supply chain findings use a structured advisory ID:

```text
SUPPLY-CHAIN|TYPOSQUAT|INSTALL_SCRIPT
SUPPLY-CHAIN|DEPENDENCY_CONFUSION
SUPPLY-CHAIN|KNOWN_MALWARE
```

The signals that contributed to the finding are encoded in the ID so they can be displayed in the UI and filtered programmatically.

## Disabling supply chain analysis

Set `supply_chain_enabled: false` in your policy's sources configuration:

```json
"sources": {
  "supply_chain_enabled": false
}
```

This is not recommended for production repositories.
