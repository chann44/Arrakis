## Scan lifecycle

Every scan goes through a defined status progression:

```text
queued → running → success
                 → failed
```

| Status | Meaning |
|---|---|
| `queued` | Job is in the queue, waiting for a worker |
| `running` | Worker has picked up the job and scanning is in progress |
| `success` | Scan completed, findings persisted |
| `failed` | Scan encountered an unrecoverable error |

The scan run record stores the trigger type, policy used, finding counts by severity, start time, and finish time.

## Triggering a scan

**Manual scan:**
```text
POST /v1/github/repositories/{repoID}/scans/run
```

**Scheduled scan:** Handled automatically by the Scheduler based on policy trigger configuration.

**Policy scan (all repos):**
```text
POST /v1/policies/{policyID}/scans/run
```

## What happens during a scan

### 1. Dependency loading

The scanner loads all `RepositoryDependency` records for the repo, then expands the transitive graph via `expandDependenciesWithGraph()` — a BFS traversal of `DependencyVersionDependency` edges to depth 4.

The result is a flat list of all packages (direct and transitive) with their resolved versions and metadata.

### 2. Advisory queries

Based on the policy's source configuration, Arrakis queries advisory databases. Queries run against the full expanded set — transitive packages at depth 3–4 are scanned the same as your direct dependencies.

**OSV batching:** OSV queries are batched at 64 packages per API call. A repo with 500 total packages makes ~8 API calls.

### 3. Supply chain analysis

`addSupplyChainFlags()` runs against **direct dependencies only** (not transitive). This is intentional — supply chain signals like install scripts and typosquats are most meaningful for packages you've explicitly chosen to depend on.

### 4. AI analysis (optional)

If `sast.ai_enabled` is set in the policy, Arrakis spawns a Docker container:

```bash
docker run --rm \
  --memory=512m \
  --cpus=1.0 \
  --cap-drop=ALL \
  --read-only \
  --security-opt=no-new-privileges \
  arrakis-ai-analyzer:local
```

The dependency list is written to the container's stdin as JSON. The container performs semantic analysis and writes findings to stdout. Results are merged into the main finding set.

The AI analyzer has a configurable timeout (default 180 seconds). If the container does not respond within the timeout, the scan continues without AI findings.

### 5. Normalization and deduplication

All findings from all sources are normalized to:

```go
type normalizedFinding struct {
  Manager         string
  Registry        string
  PackageName     string
  ResolvedVersion string
  AdvisoryID      string
  Aliases         []string
  Title           string
  Summary         string
  Severity        string
  FixedVersion    string
  ReferenceURL    string
  Sources         []string // ["osv", "ghsa", "nvd"]
}
```

The deduplication key is `{manager}|{registry}|{package_name}|{resolved_version}|{advisory_id}`. If OSV and GHSA both report the same vulnerability with different IDs, they are kept as separate findings (different `advisory_id`). If they report the same ID, they are merged with both sources listed.

### 6. Persistence

Each finding is upserted into `RepositoryScanFinding`. The database has a unique constraint on:
```text
(scan_run_id, manager, registry, package_name, resolved_version, advisory_id)
```

`RepositoryFindingOccurrence` tracks cross-scan deduplication — the same CVE in the same package across multiple scans is tracked as one occurrence with updated `last_seen_at`.

### 7. Completion

The scan run record is updated with:
- `findings_total`
- `findings_critical`
- `findings_high`
- `findings_medium`
- `findings_low`
- `status: "success"` (or `"failed"`)
- `finished_at`

## Viewing scan results

```text
GET /v1/scans                           → list all scan runs
GET /v1/scans/{scanID}                  → scan details + findings
GET /v1/scans/{scanID}/ai/logs/stream   → stream AI analyzer logs (SSE)
```

## Scan logs

Arrakis writes structured logs for each scan. If ClickHouse is configured, logs are stored there for analytics. Otherwise they're written to stdout.

Logs include:
- Dependency count loaded
- Advisory source response times
- Supply chain signals detected
- Finding counts before deduplication
- Total scan duration
