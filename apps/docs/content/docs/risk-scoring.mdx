title: Risk Scoring
description: How Arrakis maps advisory data and supply chain signals to severity levels.
---

## Two scoring systems

Arrakis has two distinct scoring mechanisms that produce findings:

1. **CVE severity** — Derived from CVSS scores in advisory databases (OSV, GHSA, NVD)
2. **Supply chain score** — Computed from behavioral signals on package metadata

Both produce findings with the same `severity` field (`critical`, `high`, `medium`, `low`), but through different paths.

## CVE severity mapping

When a vulnerability is found in an advisory database, Arrakis uses the CVSS score to determine severity:

| CVSS Score | Severity |
|---|---|
| 9.0 – 10.0 | Critical |
| 7.0 – 8.9 | High |
| 4.0 – 6.9 | Medium |
| 0.1 – 3.9 | Low |

If a CVE has multiple CVSS scores (e.g., CVSSv2 and CVSSv3), Arrakis uses the highest severity. If no CVSS score is present, the severity from the advisory record's `database_specific` field is used as a fallback.

## Supply chain score thresholds

Supply chain signals produce a cumulative integer score per package. That score maps to severity:

| Score | Severity |
|---|---|
| ≥ 50 | Critical |
| ≥ 30 | High |
| ≥ 15 | Medium |
| < 15 | Not reported |

## Signal weights reference

| Signal | Score |
|---|---|
| Known malware | +50 |
| Dependency confusion | +35 |
| Typosquat (distance > 1) | +30 |
| Typosquat (distance = 1) | +40 |
| Version anomaly | +30 |
| Install script | +25 |
| Missing lockfile | +20 |
| Abandoned with vulnerability | +15 |
| Missing repository | +10 |
| Single maintainer | +5 |

Signals stack additively. A package that is a typosquat **and** has an install script **and** no repository would score 30 + 25 + 10 = **65 → Critical**.

## Deduplication and occurrence tracking

The same finding (same package, same version, same advisory ID) can appear across multiple scans. Arrakis deduplicates via `RepositoryFindingOccurrence`:

- `first_seen_at` — When the finding was first discovered
- `last_seen_at` — When the finding was most recently confirmed
- `status` — `open` (active) or `resolved` (manually closed)

A finding that was resolved but reappears in a subsequent scan will have its `last_seen_at` updated and status reset to `open`.

## Finding counts on scan runs

Each completed scan run stores aggregated counts for fast dashboard queries:

```json
{
  "findings_total": 42,
  "findings_critical": 2,
  "findings_high": 8,
  "findings_medium": 19,
  "findings_low": 13
}
```

These counts reflect findings persisted from the scan — they do not include findings from previous scans that were not re-detected.

## Multi-source findings

When the same vulnerability is detected by multiple sources (e.g., OSV and GHSA both report `CVE-2021-23337` in lodash), Arrakis merges them into a single finding with a `sources` array:

```json
{
  "advisory_id": "GHSA-35jh-r3h4-6jhm",
  "aliases": ["CVE-2021-23337"],
  "severity": "high",
  "sources": ["osv", "ghsa"]
}
```

The finding is stored once — the severity is taken from the highest-scoring source.
