## Overview

Arrakis queries multiple advisory databases per scan and deduplicates findings across them. Each source can be enabled or disabled per policy.

The same CVE often appears in OSV, GHSA, and NVD with different identifiers and slightly different metadata. Arrakis normalises across all three and merges them into a single finding with a `sources` array indicating which databases detected it.

## OSV (Open Source Vulnerabilities)

The default and most comprehensive source. [OSV](https://osv.dev) is a distributed vulnerability database covering npm, pip, Go, Cargo, Maven, RubyGems, and many more ecosystems.

Arrakis queries the OSV batch API — up to **64 packages per request** — for efficiency:

```json
POST https://api.osv.dev/v1/querybatch
{
  "queries": [
    { "package": { "name": "lodash", "ecosystem": "npm" }, "version": "4.17.20" },
    { "package": { "name": "requests", "ecosystem": "PyPI" }, "version": "2.27.0" }
  ]
}
```

Each OSV advisory returns:
- `id` (e.g., `GHSA-xxxx-xxxx-xxxx`)
- `aliases` (e.g., `["CVE-2021-23337"]`)
- `summary` and `details`
- `severity` (CVSS score)
- `fixed` version
- `reference_url`

**Enabled by default.** Set `osv_enabled: false` in policy sources to disable.

## GHSA (GitHub Security Advisories)

GitHub's curated advisory database. Particularly strong for npm, pip, and GitHub-hosted packages. Arrakis uses GHSA data to enrich findings with additional context and aliases not always present in OSV.

Requires a GitHub token (configured via the GitHub App installation — no separate API key needed).

**Enabled by default.** Set `ghsa_enabled: false` to disable.

## NVD (National Vulnerability Database)

The US NIST CVE database. NVD has the broadest CVE coverage but requires an API key for production usage to avoid rate limiting.

```bash
NVD_API_KEY=your-key-here
```

NVD is used primarily for enrichment — adding CVSS scores, CWE classifications, and references to findings already detected by OSV or GHSA.

**Disabled by default.** Set `nvd_enabled: true` in policy sources and provide `NVD_API_KEY`.

## govulncheck

For Go projects, Arrakis can run [govulncheck](https://pkg.go.dev/golang.org/x/vuln/cmd/govulncheck) — the official Go vulnerability checker. Unlike database lookups, govulncheck performs call-graph analysis to determine whether vulnerable code paths are actually reachable in your application.

This means fewer false positives: a vulnerability in a package you import but don't call won't be reported.

**Disabled by default.** Set `govulncheck_enabled: true` in policy sources.

## Supply Chain Analysis

Not a traditional advisory database — this is Arrakis's behavioral analysis layer. It detects risks in packages that may not have any published CVE yet.

See [Supply Chain Analysis](/docs/supply-chain-analysis) for the full signal list and scoring model.

**Enabled by default.** Set `supply_chain_enabled: false` to disable.

## Custom advisory sources

Add any number of custom advisory feeds alongside the built-in sources:

```json
"custom_sources": [
  {
    "name": "Internal Security Feed",
    "url": "https://security.internal/advisories.json",
    "format": "osv",
    "auth_header": "Bearer ${INTERNAL_TOKEN}"
  },
  {
    "name": "Partner CVE Feed",
    "url": "https://partner.example.com/nvd-feed.json",
    "format": "nvd",
    "auth_header": "Basic dXNlcjpwYXNz"
  }
]
```

| Field | Description |
|---|---|
| `name` | Display name for the source |
| `url` | URL of the advisory feed (fetched at scan time) |
| `format` | `osv` or `nvd` |
| `auth_header` | Full `Authorization` header value (optional) |

Custom sources are merged with built-in sources before deduplication. A finding from a custom source appears with `source: "custom"` in the finding's sources array.

## Registry-first mode

When `registry_first: true` is set in policy sources, Arrakis checks the package registry for known-bad version signals before querying advisory databases. This can surface risks for packages that are flagged at the registry level but may not yet have a formal advisory.

```json
"sources": {
  "registry_first": true,
  "registry_max_age_days": 3
}
```

`registry_max_age_days` controls how long registry metadata is considered fresh before being re-fetched. Defaults to `7`.
