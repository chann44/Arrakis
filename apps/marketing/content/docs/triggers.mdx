---
title: Triggers
description: Control when Arrakis scans run — on a schedule, manually, or on git events.
---

## Trigger types

| Type | Description |
|---|---|
| `schedule` | Cron-based automated scanning |
| `manual` | User-initiated via API or dashboard |
| `push` | On git push (planned) |
| `pull_request` | On pull request (planned) |

A policy can have multiple triggers of different types.

## Schedule trigger

The Scheduler service evaluates all active policy triggers every **30 seconds**. When a cron expression fires, it:

1. Creates a `RepositoryScanRun` record with `trigger: schedule`
2. Enqueues a scan job on the `scans` queue
3. Sets a Redis deduplication key to prevent duplicate runs in the same minute

```json
{
  "type": "schedule",
  "cron": "0 2 * * *",
  "timezone": "America/New_York"
}
```

### Cron format

Standard 5-field cron: `minute hour day-of-month month day-of-week`

| Expression | Meaning |
|---|---|
| `0 2 * * *` | Every day at 2:00 AM |
| `0 */6 * * *` | Every 6 hours |
| `0 9 * * 1` | Every Monday at 9:00 AM |
| `0 0 * * 0` | Weekly on Sunday at midnight |

### Timezone

Each trigger supports an independent timezone. The Scheduler converts the cron expression to the specified timezone before evaluation. Use any [IANA timezone name](https://en.wikipedia.org/wiki/List_of_tz_database_time_zones):

```json
"timezone": "America/New_York"
"timezone": "Europe/London"
"timezone": "Asia/Tokyo"
"timezone": "UTC"
```

### Deduplication

The Scheduler prevents the same policy from triggering duplicate scans within the same minute using a Redis guard key:

```
scan:schedule:{policyID}:{repoID}:{time_slot}
```

TTL: 2 minutes. This means even if the Scheduler restarts mid-minute, the scan will not be double-queued.

## Manual trigger

Scans can be triggered via the dashboard or API at any time:

```
POST /v1/github/repositories/{repoID}/scans/run
```

```
POST /v1/policies/{policyID}/scans/run
```

The policy-level endpoint queues a scan for all repositories assigned to that policy.

## Job queue

Scans are processed by the Worker service using [asynq](https://github.com/hibiken/asynq) — a Redis-backed task queue.

| Queue | Concurrency | Lock timeout |
|---|---|---|
| `scans` | 8 workers | 30 minutes |
| `dependencies` | 10 workers | 45 minutes |

If a scan job fails, asynq retries it up to **4 times** with exponential backoff. If all retries are exhausted, the scan run is marked `failed` with the error message stored on the record.

### Concurrent scan protection

A distributed lock (`scan:repo-lock:{repoID}`) prevents two concurrent scans of the same repository. If a scan is already in progress when a new job is picked up by a worker, the new job will fail immediately — the existing scan continues unaffected.
