## Overview

Arrakis operates in four phases: **Connect → Sync → Scan → Report**. Each phase is handled by a different service, coordinated through a Redis-backed job queue.

```
GitHub Repo
    │
    ▼
┌─────────────────────────────────────────────┐
│  Phase 1: Connect                           │
│  POST /v1/github/repositories/{id}/connect  │
│  Creates Repository record                  │
│  Queues DependencySync job                  │
└─────────────────────┬───────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────┐
│  Phase 2: Dependency Sync (Worker)          │
│  Lists repo tree via GitHub App             │
│  Parses manifest files                      │
│  Resolves registry metadata                 │
│  Builds dependency graph (depth 4)          │
└─────────────────────┬───────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────┐
│  Phase 3: Scan (Worker)                     │
│  Loads dependency graph                     │
│  Queries advisory sources in parallel       │
│  Runs supply chain analysis                 │
│  Runs AI analyzer (optional)                │
│  Deduplicates + persists findings           │
└─────────────────────┬───────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────┐
│  Phase 4: Report (API)                      │
│  GET /v1/scans/{scanID}                     │
│  Findings by severity                       │
│  Push to integrations                       │
└─────────────────────────────────────────────┘
```

## Phase 1: Repository connection

When you connect a repository, Arrakis:

1. Stores repository metadata (name, default branch, visibility) from the GitHub API
2. Persists a `Repository` record linked to your user
3. Immediately enqueues a `DependencySync` job on the `dependencies` queue

The API responds before the sync completes — the repository status will show as `syncing` until the worker finishes.

## Phase 2: Dependency sync

The worker picks up the `DependencySync` job and calls `SyncRepositoryDependencies()`:

**Distributed lock** — Arrakis acquires a per-repository lock (45 min timeout) to prevent concurrent syncs racing each other.

**Manifest discovery** — The GitHub App token is used to list the repository tree recursively. Files matching known patterns are identified:

| Manifest | Manager |
|---|---|
| `package.json`, `yarn.lock`, `pnpm-lock.yaml` | npm |
| `requirements.txt`, `pyproject.toml`, `poetry.lock` | pip |
| `go.mod`, `go.sum` | Go |

**Parsing** — Each manifest file is downloaded and parsed:

- `package.json`: JSON parsing of `dependencies`, `devDependencies`, `peerDependencies`
- `requirements.txt`: Line-by-line regex extraction of package specifiers
- `go.mod`: Module directive and require block parsing

**Registry resolution** — For each package, Arrakis queries the appropriate registry to get exact version metadata:

- npm: `https://registry.npmjs.org/{package}`
- PyPI: `https://pypi.org/pypi/{package}/json`
- Go: `https://proxy.golang.org`

Metadata collected includes: creator, description, license, homepage, repository URL, release date, all known versions, and whether an install script is present.

**Graph persistence** — Dependency edges are stored as `DependencyVersionDependency` records. Transitive dependencies are followed recursively up to **depth 4** and deduplicated by composite key `{manager}|{registry}|{name}|{version}`.

## Phase 3: Vulnerability scan

`RunRepositoryScan()` orchestrates the full scan pipeline:

**Load graph** — All `RepositoryDependency` records for the repo are loaded, then `expandDependenciesWithGraph()` performs a BFS traversal of `DependencyVersionDependency` edges up to depth 4, building the complete set of packages to scan.

**Advisory queries** (run per policy configuration):

- **OSV**: Batch API — up to 64 packages per request. Maps `{name, ecosystem, version}` to advisory records with CVE IDs, aliases, severity, and fixed versions.
- **GHSA**: GitHub Security Advisory API for GitHub-hosted packages.
- **NVD**: National Vulnerability Database for CVE enrichment.
- **Custom feeds**: User-provided advisory URLs in OSV or NVD JSON format.

**Supply chain analysis** — `addSupplyChainFlags()` runs independently of advisory sources. It evaluates each direct dependency against a set of behavioral signals (see [Supply Chain Analysis](/docs/supply-chain-analysis)).

**AI analysis** (optional) — A sandboxed Docker container is spawned with the full dependency list. It performs semantic analysis and returns findings with confidence scores. Results are merged into the main finding map.

**Deduplication** — Findings are normalized to a composite key:
```
{manager}|{registry}|{package_name}|{resolved_version}|{advisory_id}
```

Multiple sources detecting the same finding are collapsed — the finding record stores an array of source names (e.g., `["osv", "ghsa", "nvd"]`).

**Persistence** — Findings are upserted into `RepositoryScanFinding`. Occurrence tracking via `RepositoryFindingOccurrence` maintains `first_seen_at` and `last_seen_at` across scans.

## Phase 4: Reporting

Findings are available immediately via the API once the scan completes:

```
GET /v1/scans/{scanID}         → findings for a specific scan
GET /v1/findings               → all findings across all repos
GET /v1/findings/{findingID}   → single finding with full source detail
```

Scan status progresses through: `queued` → `running` → `success` / `failed`.

Finding counts per severity are stored on the scan run record for fast dashboard queries without loading all findings.
