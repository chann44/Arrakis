## Why transitive dependencies matter

Most vulnerability scanners only check your **direct** dependencies — the packages you explicitly listed in `package.json` or `go.mod`. But the real attack surface is your transitive graph.

A typical Node.js application with 50 direct dependencies will have 500–2,000 transitive ones. Most CVEs and supply chain attacks target packages at depth 2–4, not depth 1. Checking only your direct deps misses the majority of real risk.

Arrakis traverses the full graph to **depth 4** for every scan.

## How the graph is built

### Manifest discovery

When a repository is synced, Arrakis lists the full repository tree via the GitHub App API and identifies manifest files by name:

| File | Manager | Registry |
|---|---|---|
| `package.json` | npm | npmjs.org |
| `yarn.lock` | npm | npmjs.org |
| `pnpm-lock.yaml` | npm | npmjs.org |
| `requirements.txt` | pip | pypi.org |
| `pyproject.toml` | pip | pypi.org |
| `poetry.lock` | pip | pypi.org |
| `go.mod` | go | proxy.golang.org |
| `go.sum` | go | proxy.golang.org |

Multiple manifest files in the same repository are all processed — a monorepo with a Go backend and a Node.js frontend will have both graphs built and merged.

### Registry resolution

For each package found in a manifest, Arrakis queries the registry to resolve exact metadata:

```
package.json: "express": "^4.18.0"
                      │
                      ▼
  registry.npmjs.org/express
                      │
                      ▼
  resolved version: 4.18.2
  metadata: creator, license, description,
            repository URL, release date,
            all known versions,
            has_install_script: true/false
```

Resolved metadata is stored in `DependencyPackageVersion` and reused across repositories that share the same package version — reducing registry API load.

### Transitive traversal

Arrakis follows the `dependencies` field of each resolved package version to build edges in the graph:

```
your app
  └── express@4.18.2         (depth 1)
        └── body-parser@1.20 (depth 2)
              └── bytes@3.1  (depth 3)
```

These edges are stored as `DependencyVersionDependency` records. The traversal is bounded at **depth 4** to prevent exponential blowup in large graphs.

**Deduplication** — If the same package version appears at multiple depths (common in npm), it's tracked once. The deduplication key is `{manager}|{registry}|{name}|{version}`.

### Graph schema

```
RepositoryDependency          — direct dep declaration in a repo
DependencyPackage             — unique package identity (name + manager + registry)
DependencyPackageVersion      — specific version with full metadata
DependencyVersionDependency   — edge between two version nodes (from → to)
```

## How the graph is used during scanning

When a scan runs, `expandDependenciesWithGraph()` performs a BFS from all direct dependencies:

```go
// pseudocode
queue := direct_dependencies
visited := {}
depth := 0

while queue is not empty and depth < 4:
  for each dep in queue:
    if dep not in visited:
      visited.add(dep)
      children = load_edges(dep.version_id)
      queue.extend(children)
  depth++

return visited
```

All packages in the resulting set — direct and transitive — are sent to advisory sources for scanning.

## Forcing a re-sync

By default, if a package version is already in the database, its metadata is not re-fetched from the registry. To force a full re-sync:

```
POST /v1/github/repositories/{repoID}/deps/sync
Body: { "force": true }
```

This is useful when a package has published new metadata (maintainer changes, updated repository URLs) that you want Arrakis to pick up.
